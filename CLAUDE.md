## This repo is now the only place this app is developed

`index.html` was rewritten by merging two previously-diverged copies of Graba:

1. **This repo's own `index.html`** (the live, deployed app) — plain vanilla JS with a real, working Supabase backend: real auth, real trip/booking storage, a real Claude-powered AI concierge, and a hotel-search edge function. (At the time of this merge it also had a Travel Wallet with Paystack top-ups; the wallet was later removed entirely — see "Wallet removal" below.)
2. **A separately Lovable-redesigned copy** of the same app, developed in the *other* repo `unifytorise-hue/graba-global` (lowercase, a Lovable/TanStack project) — much more polished UI (Playfair Display + Inter fonts, glow effects, mobile safe-area handling, a nicer multi-thread AI chat panel called **Gabriella**, a profile overlay, a first-run onboarding walkthrough, a "pack belt" destination carousel). Its auth/wallet/trips were **100% fake** — plain `localStorage`, a `Prototype · Demo Data` badge, no real backend at all.

Those two copies were merged in branch `claude/merge-lovable-ui`: the Lovable UI became the new `index.html`, with every piece of fake `localStorage` auth/wallet/trip logic ripped out and replaced with the real Supabase wiring from the original file (same tables, same RPCs, same edge functions, same `requireAuth()` gating pattern — see Backend below). **`unifytorise-hue/graba-global` (lowercase) is now superseded and frozen** — do not make product changes there; it was a UI-only prototype and has no real backend of its own. All future work, on both UI and backend, happens in this repo (`GRABA-GLOBAL-`, note the trailing dash) only. If you're ever unsure which repo to edit, it's this one.

## What this is

**Graba** ("Making Travel Simple") — a travel booking app: search a destination on an interactive 3D globe, browse hotels on a live map, bundle a flight + hotel + transfers into one package, and pay by card at booking time (charged via Paystack — see "Payments" below), or ask **Gabriella** (the AI concierge) to help plan a trip. All markup, CSS, and JS still lives in one `index.html` (no build step, bundler, or framework), but auth and money live in a real backend rather than in-memory state — see "Backend" below. There is no test suite; the app is opened directly in a browser or served statically.

External dependencies loaded from CDN in `index.html`: Google Fonts (Playfair Display + Inter + Outfit), Leaflet 1.9.4 + leaflet.markercluster (destination map/tiles), the Supabase JS SDK v2, and per-hotel Unsplash/generated-SVG photos with graceful fallbacks (`photoTag()`).

Flight/hotel inventory is still procedurally generated mock data (see Architecture below). The plan is Sabre for flights (Bargain Finder Max for search, EnhancedAirBook for PNR creation) and Booking.com Partner/Demand API for hotels — chosen over routing hotels through Sabre too, since Sabre's hotel content/booking certification is a separate, weaker path than its flight product. Both are blocked on the respective partner accounts; see the edge functions below for the current seams.

## Wallet removal (regulatory/security decision)

**The Travel Wallet feature has been removed entirely** — the product owner decided a stored-value wallet was an unnecessary regulatory/security risk (in South Africa, general-purpose stored value can brush up against e-money/deposit-taking rules; it also meant every wallet-crediting bug was a direct financial-loss bug). While removing it, it also came to light that the previous "pay by card" option **never actually charged anything** — `book_trip`'s `p_payment_method` was purely a label, and no Paystack charge was ever initiated for the card-covered portion of a booking. Both issues are fixed together:

- The wallet UI (chips, the wallet sheet, the payment-method chooser in the booking modal) is gone from `index.html`, along with `topUpWallet()`, `loadWalletAndTrips()` (renamed/simplified to `loadTrips()`, bookings-only), and everything else wallet-shaped.
- **Card payment at booking time is now real.** `bookBtn.onclick` no longer calls `book_trip` and creates a booking instantly — it calls the new `create-booking-payment` edge function, which starts a real Paystack transaction and redirects the browser to Paystack's hosted checkout. The booking is only ever created **after** Paystack confirms the charge, server-side, by `paystack-webhook` calling the new `admin_create_booking` RPC — mirroring the same "init transaction client-side, webhook does the trusted write" pattern the wallet top-up flow used.
- **`wallets` and `wallet_transactions` tables were deliberately NOT dropped** — they remain in the schema, RLS policies untouched, just dormant/unused. Cheap to resurrect later if the product decision changes; dropping tables for what's fundamentally a frontend/flow change would have been unnecessarily destructive.
- **`book_trip`, `top_up_wallet`, and `admin_top_up_wallet` were deliberately left alone** — dead/unused in the database, not cleaned up, out of scope for the removal. Do not call `book_trip` from the frontend again; it's the RPC responsible for the old "instant unpaid booking" bug.

## Backend (Supabase)

Auth and bookings are backed by a dedicated Supabase project **"Graba Global"** (`mpkklamqfjwfuciuuxae`, eu-west-1) — separate from the unrelated AestheticFlow CRM project in the same org. Schema:

- `profiles` — 1:1 with `auth.users`, auto-created by an `on_auth_user_created` trigger (`handle_new_user()`) on signup.
- `wallets` / `wallet_transactions` — **dormant, unused** since the wallet removal above. Left in place, not dropped.
- `bookings` — one row per confirmed trip; flight/hotel/transfer details are stored in a `details jsonb` column shaped to match a real Sabre response, so swapping mock data for live Sabre PNR data later won't need a schema change.

All tables are RLS-scoped to `auth.uid()` (SELECT only from the client — no direct client-side booking mutation). Booking creation goes through `security definer` RPCs instead (`anon`/`authenticated` execute is explicitly revoked by default and re-granted only where intended, since Postgres/Supabase grants `EXECUTE` to `anon` by default on new functions):

- `book_trip(...)` — **dead/unused.** The old direct-booking RPC; left in the database untouched (see Wallet removal above) but no longer called from `index.html`.
- `top_up_wallet(...)` / `admin_top_up_wallet(...)` — **dead/unused**, wallet-era RPCs left in the database untouched.
- `admin_create_booking(p_user_id, p_destination_city, p_destination_country, p_traveller_name, p_nights, p_total_amount, p_details, p_payment_ref)` — the real replacement for `book_trip`. `service_role`-only (not `authenticated`, not `anon`) — only callable by the `paystack-webhook` edge function after it has verified a real Paystack payment. Generates its own `booking_ref` (`GRB-XXXXXXXX`), inserts with `payment_method='card'`, and merges `p_payment_ref` into `details`. Does not touch `wallets`/`wallet_transactions` at all. See Payments below.
- `add_insurance(p_booking_id)` — idempotently adds R150 to an existing booking's total and flags `details->>'insurance'`.

In `index.html`, `loadTrips()` re-fetches bookings after every mutating RPC call and re-renders; there's no realtime subscription.

**Auth is on-demand, not a startup gate.** `#authOverlay` (email/password sign-in/sign-up, toggled via a `.on` CSS class rather than `display`) never shows automatically — anyone can browse the globe, search, view hotel/flight details, and open the package builder without an account. The single gate is `requireAuth(message)` (defined near the top of `index.html`'s main `<script>`, alongside the `sb` client): returns `true` immediately if `currentUser` is set, otherwise calls `openAuth(message)` to show `#authOverlay` with a contextual subtitle and returns `false`; callers just `return` when it's `false` and the user can retry the same action once signed in. It's used at exactly the points that need a real account: opening trips (`tripsBtnGlobe`/`tripsBtnMap`), opening Gabriella (the AI concierge FAB), and — the one that actually matters — the "Book with Graba" button itself. `sb.auth.onAuthStateChange` (registered in the auth `<script>` block near the end of the file, next to the sign-in/up form) never force-shows the overlay on a null session; it just resets the trips display to empty. The header's `#authBtnGlobe` button (globe view, top-right) doubles as both the sign-in entry point and, once signed in, shows the account's first name and opens the same overlay in "account view" (name/email + sign out) — there's no separate burger menu in this UI.

Client-side `SUPABASE_URL`/`SUPABASE_ANON_KEY` in `index.html` are the anon key — safe to expose, since every table relies on RLS + the RPCs above rather than trusting the client.

### AI concierge — "Gabriella" (Claude)

`concierge-chat` is a chat-based AI travel concierge — a Claude tool-use loop (Anthropic Messages API) over Graba's own capabilities. In the UI it's branded **Gabriella**, with a floating action button, a multi-thread chat panel (`#gabrielPanel`/`#gabrielSheet`, `initGabriel()` IIFE near the end of the main `<script>`), and a thread list (`#gThreadsPane`/`#gThreadList`).

- **Two message histories per thread, deliberately kept separate**: `t.messages` (simple `{role, content:string}` pairs — used to render the chat UI and persisted to `localStorage` under `graba_gabriel_threads_v1`/`graba_gabriel_active_v1`, since which threads exist and their titles is just local UI bookkeeping, not account data) and `t.apiMessages` (the raw Claude Messages API history, which can include `tool_use`/`tool_result` content blocks) sent to and returned from `concierge-chat` so multi-turn tool use keeps working within a thread without leaking those blocks into the rendered chat. The actual chat completions always come from the real edge function — nothing about Gabriella's replies is mocked.
- Opening Gabriella (the FAB) is gated by `requireAuth()`; sending a message additionally checks for a live Supabase session (`sb.auth.getSession()`) since `concierge-chat` requires a bearer token.
- Not a streaming endpoint — `concierge-chat` returns a final `{reply, messages}` once Claude's tool-use rounds finish, so `send()` in the Gabriella IIFE shows a typing indicator and then renders the whole reply at once (there's no token-by-token stream to consume, despite the UI supporting one).
- **Tools exposed to Claude**: `search_flights` and `search_hotels` (server-to-server HTTP calls to the sibling `sabre-flight-search`/`booking-hotel-search` edge functions, using the service role key — they inherit those functions' own mock-data fallback behavior), and `propose_booking`. `get_wallet_balance` (which queried the now-dormant `wallets` table) has been **removed entirely** — both the tool schema sent to Claude and its handler branch — since the wallet feature no longer exists; Gabriella no longer claims it can check a wallet balance.
- **Guardrail — do not relax without discussion**: `propose_booking` is a pure pass-through that returns a structured proposal for the frontend to render; it does **not** call `book_trip`. Nothing in this function can spend money or create a real booking — the user still has to go through the existing package-builder modal and tap Confirm. This is deliberate: an LLM autonomously committing bookings/payments with no human confirmation step is not a safe default. The system prompt also explicitly tells Claude never to claim something is booked/charged.
- **Confidence**: HIGH on the Claude/Anthropic API shape itself (Anthropic's own documented API, unlike the Sabre/SerpApi/LiteAPI work), but this has **not** been tested end-to-end from this sandbox (no live Anthropic API access here either) — verify a real conversation once `ANTHROPIC_API_KEY` is set.
- Requires an `ANTHROPIC_API_KEY` edge function secret (console.anthropic.com) and, optionally, `CLAUDE_MODEL` to override the default (`claude-sonnet-5`).

### Payments (Paystack — active, now charges at booking time; Stripe is dormant)

Real money moves through **Paystack**, wired into the booking flow (`bookBtn.onclick` in `index.html`; the post-redirect handler is `handlePaystackRedirect()`). This used to be wired into a Travel Wallet top-up flow instead — see "Wallet removal" above for why that changed. The business went with Paystack over Stripe (better South African merchant support), so this superseded the earlier Stripe integration. Two edge functions:

- **`create-booking-payment`** (`verify_jwt: true`, source at `supabase/functions/create-booking-payment/index.ts`) — called by a signed-in user with the full booking payload (`{total_amount, destination_city, destination_country, traveller_name, nights, details}`); calls Paystack's `POST /transaction/initialize` (amount in kobo/cents, `ZAR`, `callback_url: ${SITE_URL}?booking=paystack`), tagging the transaction with `metadata.kind='booking'`, `metadata.user_id`, and `metadata.booking` (the fields needed to create the booking once payment confirms), and returns `{url: authorization_url}` for the frontend to redirect to. Requires `PAYSTACK_SECRET_KEY` and the user to have an email on file (Paystack requires one); falls back to a clear error if either is missing, since there's no safe fallback for real payments the way there is for search. No booking row is created by this function — it only starts the payment.
- **`paystack-webhook`** (`verify_jwt: false` — deliberately, Paystack calls it directly with no Supabase session, authenticating instead via HMAC-SHA512 of the raw body against the `x-paystack-signature` header, verified manually via Web Crypto since there's no Deno-friendly Paystack SDK used here; source at `supabase/functions/paystack-webhook/index.ts`) — the **only** path that actually creates a booking (or, historically, credited a wallet). On `charge.success`, if `metadata.booking` is present it calls `admin_create_booking` via the service role key using the fields from `metadata.booking` plus `metadata.user_id`; otherwise (no booking metadata — e.g. a stray pre-removal wallet top-up transaction) it just logs and acknowledges, rather than erroring. Deduplicates via a `paystack_events` table (transaction reference as primary key) since Paystack can redeliver webhooks.
- Requires edge function secrets: `PAYSTACK_SECRET_KEY` (both functions — also used to verify the webhook signature), optionally `SITE_URL` (defaults to the GitHub Pages URL) for the `callback_url`. **Whether `PAYSTACK_SECRET_KEY` is actually set/activated in this project has not been directly verified from this environment** — confirm in the Supabase dashboard (Edge Functions → Secrets) before relying on this in production; without it, `create-booking-payment` returns a clean 500 rather than silently failing.
- **Redirect handling**: Paystack uses a single `callback_url` for both success and failure (it appends `?trxref=...&reference=...`), unlike Stripe's separate success/cancel URLs — there's no explicit "cancelled" signal in the URL. `handlePaystackRedirect()` treats the presence of a `reference`/`trxref` query param on `?booking=paystack` as "a booking payment was attempted," then polls `loadTrips()` a few times (webhook delivery can lag the browser redirect) looking for the new booking (matched by `details.paymentRef === reference`) and shows the "Trip booked" confirmation screen once found; if it never shows up after retrying, it opens My Trips instead so the user isn't stuck on a blank screen (the booking will still appear there once the webhook lands).
- **Confidence**: moderate-high on Paystack's Transaction Initialize API and webhook signature scheme (well-documented, stable REST API) — higher than Sabre/Booking.com/SerpApi, but not at the same certainty level as the Claude/Anthropic integration. **Untested against a real Paystack account** from this environment (no live Paystack access here). Verify a real test-mode payment once `PAYSTACK_SECRET_KEY` is confirmed set/activated, and the webhook URL (`https://mpkklamqfjwfuciuuxae.supabase.co/functions/v1/paystack-webhook`) is registered in the Paystack dashboard.
- **`create-paystack-transaction`** (the old wallet top-up initializer) **is still deployed but no longer called from `index.html`** — left in place rather than deleted, same as the Stripe functions below, in case it's useful if the wallet is ever resurrected.
- **Stripe's `create-checkout-session`/`stripe-webhook` functions and `stripe_events` table are still deployed but unused** — left in place rather than deleted in case Stripe is revisited later.

### Booking modal — presentation-layer reskin (no backend/payment/reservation logic changed)

The package-builder modal (`#modal`, opened via `openPackageModal()`) was reorganized for clarity — applying generic, industry-standard booking-flow UX (itemized always-visible pricing, a jump-nav rather than a gated wizard, a trust cue before charging a real card), not a copy of any specific competitor's design:

- The step-dots (`#modalSteps`) are a **jump-nav over one continuous scrollable view, not a gated wizard** — every section (`MODAL_STEP_IDS`: `stepNights`, `stepFlight`, `stepTransfers`, `stepGuest`, `stepReview`) stays visible and editable regardless of which dot is "active"; tapping a dot just scrolls to it (`updateModalSteps()`'s click handler, unchanged from before).
- `#priceBreakdown` shows live per-line pricing (stay / flight / transfers) above the total, populated by `updateTotal()` from the same `currentTotal()` this always used — no new pricing logic, just more of the existing numbers rendered.
- New `#stepReview` section (`#rvDest`/`#rvStay`/`#rvFlight`/`#rvTransfers`/`#rvGuest`) — a one-glance summary of exactly what's being booked, also populated by `updateTotal()`.
- `#stepPayment` (the old non-interactive "Credit / Debit Card" line) became `#paymentTrust`, moved to sit directly above `#bookBtn` with explicit copy ("you'll be redirected to Paystack... your card is charged immediately, this isn't a hold") — a trust/reassurance cue right before the one action that actually spends real money.
- Added `#originInput` (free-text departure city/airport, defaults to `Johannesburg (JNB)`, stored in `originCity`) — this is **UI-only**, included in the booking's `details.originCity` sent to `create-booking-payment` so it's at least recorded, but **not yet wired to real flight search** (flights are still `genFlights()` mock either way — see below). A real airport-autocomplete/IATA lookup was deliberately not built here, out of scope for a presentation change.
- Nothing about `bookBtn.onclick`'s actual call chain changed: `requireAuth` → LiteAPI prebook (if the hotel has a real `offerId`) → `create-booking-payment` → redirect to Paystack. This was a restructure of the surrounding UI only.

### Flight/hotel edge functions

Two edge functions are deployed on the Graba Global project as seams for real inventory. **Hotels are wired into the frontend; flights are not yet.** The hotel wiring degrades safely: `fetchRealHotels()` falls back to `genHotels()` mock data on any error, so a broken call just silently reverts to generated inventory rather than breaking the map view (this is the "never zero availability" principle — see below).

- **Hotels** (`fetchRealHotels()` / `adaptHotelSearchResult()` in `index.html`, called from `showMapView()` and `searchThisArea()`) — calls `booking-hotel-search` with `{query: "<city>, <country>", nights, count, lat, lon}` (`lat`/`lon` come straight from the matched `DESTINATIONS` entry, or the map's current center when panning), adapts its `{id,name,type,pricePerNight,rating,currency,lat,lon,offerId,liteApiId}` response shape into the full internal hotel shape (`x`/`y` percentage-space coords are derived by inverting `genHotels()`'s own lat/lon formula; `scene`, `rooms`, `supplierPriority`, `profitPct` have no real-world source so stay randomised even for real hotels — the last three are Graba's own invented business-logic weights, not something any real API would provide). On any error, empty result, or exception, `showMapView()` falls back to `genHotels(currentDestKey, 60)` mock data — the map view never breaks even if LiteAPI/the edge function is down or `LITEAPI_KEY` isn't set yet.
  - **Source: LiteAPI, not SerpApi.** The edge function's data source was switched from SerpApi's `google_hotels` engine (search/price-display only, scraped, never bookable) to **LiteAPI** (https://www.liteapi.travel/), a real bookable hotel-supply API. The frontend request/response contract is unchanged — this was a same-contract backend swap, the frontend didn't need to change field names.
  - The edge function source now lives in this repo at `supabase/functions/booking-hotel-search/index.ts` (previously it existed only as a deployed function, not version-controlled). It calls LiteAPI's `GET /v3.0/data/hotels` (by lat/lon, 30km radius, capped at 20 results) to list nearby properties, then `POST /v3.0/hotels/rates` (checkin = today+7 days, checkout = checkin+nights, matching the old SerpApi function's date defaults) to price them. A hotel is only included in the response if the rates call actually returned a price for it — the function never invents a `pricePerNight`, so if pricing fails for a hotel it's dropped rather than given a fake number.
  - Requires a `LITEAPI_KEY` edge function secret (read via `Deno.env.get("LITEAPI_KEY")`) — **not yet set**. Until it is, the function returns a clean `500 {error: "LITEAPI_KEY is not configured as an edge function secret"}`, and the frontend's existing fallback-to-mock-hotels behavior kicks in exactly as it did for the missing `SERPAPI_KEY` before — hotel search still "works" (shows plausible-looking hotels), it's just not real inventory yet. **Set `LITEAPI_KEY` before hotel search returns real, bookable results.**
  - `loadMoreBelt()`'s infinite-scroll padding still uses local mock data unconditionally (deliberate — avoids burning LiteAPI quota on scroll, and blends real initial results with the app's existing "never zero availability" padding).
- **Flights** (`sabre-flight-search`) — still not called by the frontend's booking flow at all; `openPackageModal`/`shuffleFlights` use local `genFlights()` directly (there's a separate, unrelated `fetchRealFlights()` in `index.html` that calls a Lovable-prototype-only `/api/public/flights` endpoint that doesn't exist in this static deployment — it fails silently and falls back to `genFlights()`, so it's harmless, but it's dead code pointing at nothing; wiring `sabre-flight-search` for real is still open). Wire it up once Sabre credentials are sorted and this has been tested for real (see the section below).

- **`sabre-flight-search`** (v6) — does the Sabre OAuth2 token exchange, then calls Bargain Finder Max. Both request and response are now built against verified real samples:
  - **Request** (`buildBFMRequest()`) — built against a *verified real* `OTA_AirLowFareSearchRQ` sample provided directly from Sabre's own docs.
  - **Response** (`parseBFMResponse()`) — built against a *verified real* `groupedItineraryResponse` sample (Sabre's modern JSON BFM response, not the older XML-derived `OTA_AirLowFareSearchRS` shape). Walks the reference-ID chain: `itineraryGroups[].itineraries[].legs[].ref` → `legDescs[]` → `schedules[].ref` → `scheduleDescs[]` for airline/time/duration/stops, and `pricingInformation[0].fare.totalFare` for price. Also checks `messages` for `severity: "Error"` entries and surfaces them. Pass `{"debug": true}` to always get the raw response back for comparison regardless.
  - **Known gap — currency**: the verified response sample prices in USD (`totalFare.currency`), but nothing in Graba's UI does currency conversion — `fmtR()` just prefixes with "R" unconditionally. Real Sabre prices will be silently mislabeled as ZAR until a conversion step is added. `FlightOption` now carries a `currency` field so the frontend can at least detect this once wired in.
  - **Token exchange encoding** (`getSabreToken()`) — `SABRE_CLIENT_ID`/`SABRE_CLIENT_SECRET` are the raw "User ID"/"Password" from Sabre Dev Studio's Applications page (format `V1:xxxxxxxxxxxx:DEVCENTER:EXT`). Standard OAuth2 Basic auth is a single `base64(id:secret)`, but Sabre has historically documented a Sabre-specific *double*-encoding scheme for this exact credential pair — recalled with moderate confidence, not verified. The function tries single-encoding first and automatically falls back to double-encoding if rejected; whichever worked is reported back as `tokenEncodingUsed`.
  - **Endpoint path** (`SABRE_BFM_PATH`) — still unverified, defaults to a guessed `/v2.3.0/shop/flights`; override via that secret once the real path is confirmed from Sabre Dev Studio docs.
  - Requires `SABRE_CLIENT_ID`, `SABRE_CLIENT_SECRET`, and `SABRE_PCC` (your account's Pseudo City Code) as edge function secrets — falls back to mock flights (`genFlights()`-shaped) if any are missing.
  - **Origin airport**: `index.html` now has a real "Flying from" `<select>` in the package builder modal (populated from `DESTINATIONS`, defaulting to Johannesburg/`JNB` — global `originCode`/`originCity`), so this is no longer nonexistent. It's just not plumbed into an actual `sabre-flight-search` call yet, since the frontend doesn't call that function at all (see above).
  - Booking a selected flight into a PNR would additionally need EnhancedAirBook, not yet stubbed anywhere.
- **`booking-hotel-search`** (despite the name, doesn't target Booking.com) — hotel search backed by **LiteAPI** (see above for the full description; source at `supabase/functions/booking-hotel-search/index.ts`). Pass `{"debug": true}` in the request body to get the raw LiteAPI list+rates responses back alongside the mapped hotels, to check/fix the mapping against real output. **`LITEAPI_KEY` must be set manually** (Supabase dashboard → Project Settings → Edge Functions → Secrets, or `supabase secrets set LITEAPI_KEY=...`) — no MCP tool here can set edge function secrets.
  - **Single-hotel reprice mode** (added for real check-in date selection, see below) — pass `{hotelId, checkin, checkout}` (raw LiteAPI id + explicit `YYYY-MM-DD` strings) instead of `{lat, lon}` and it skips the `/v3.0/data/hotels` list call entirely, going straight to `/v3.0/hotels/rates` for just that one hotel, returning the same `{source, hotels: [...]}` shape with zero or one hotel. `checkin`/`checkout` also work as an override in the normal list-mode path when supplied — both modes share one internal `priceHotels()` function so there's no duplicated rate-fetching logic. When `hotelId` is absent, list-mode behavior is unchanged from before this mode existed.

### Real hotel reservation — LiteAPI prebook → book (wired into the payment webhook)

A completed, paid booking now attempts a **real** LiteAPI reservation, not just a real charge. The flow, end to end:

1. `booking-hotel-search` returns an additional `offerId: string | null` per hotel (additive — every previously-existing field is unchanged). `null` if no offer id could be found for that hotel's rate (never invented).
2. The frontend threads `offerId` onto the internal hotel object in `adaptHotelSearchResult()` (`index.html`). Mock/fallback hotels from `genHotels()` never have this field at all.
3. Right before `bookBtn.onclick` starts the Paystack transaction, if `activeHotel.offerId` is present it calls the new **`liteapi-prebook`** edge function (`supabase/functions/liteapi-prebook/index.ts`, `verify_jwt: true`) with `{offerId}`, which calls LiteAPI's `POST /v3.0/rates/prebook` and returns `{prebookId, expiresAt, price, raw}`. Mock hotels (no `offerId`) skip this entirely and book exactly as before — they were never going to be a real reservation. Any prebook failure is logged (`console.warn`) and booking proceeds anyway rather than blocking the user's payment.
4. `bookBtn.onclick` also splits the single `travellerName`/`travellerInput` field into a best-effort `{firstName, lastName}` (first space is the split point — no new form fields were added) and sends both `prebookId` and `holder` to `create-booking-payment`.
5. `create-booking-payment` stashes `prebookId` and `holder` (email always taken from its own verified `user.email`, never trusted from the client) into `metadata.booking`, unchanged from there through Paystack.
6. `paystack-webhook`, after `admin_create_booking` successfully creates the `bookings` row, reads `metadata.booking.prebookId`/`holder` back out and — inline, in the same function (`bookLiteApiHotel()`) — calls LiteAPI's actual booking endpoint (`POST https://book.liteapi.travel/v3.0/rates/book`, note: a **different host** than the rest of LiteAPI's API) with `{holder, guests, payment: {method: "ACC_CREDIT_CARD"}, prebookId}`.
   - **Success** → `bookings.details` gets `supplierConfirmed: true` + `supplierBookingRef`.
   - **Failure, or no `prebookId` at all** (mock hotel, or the client-side prebook step failed/was skipped) → `bookings.details` gets `supplierBookingFailed: true` + `supplierBookingFailureReason`. **This never fails the webhook or touches the booking/payment** — the charge already happened and the internal booking row already exists by this point; that must stand regardless of supplier outcome.

**CONFIDENCE — LOW-MODERATE, UNVERIFIED, read before trusting this.** This sandbox never had live internet access to LiteAPI's actual documentation or a real LiteAPI account, so:
- The exact field name for "the offer id" inside a `POST /v3.0/hotels/rates` rate object was **not directly confirmed**. The only reference on disk (`liteapi-hotels.server.ts` in the sibling `graba-global-lovable` prototype repo) only ever read `offerRetailRate`/`rates[].retailRate` for *price* — it has no offer/rate-id field modeled at all, so there was no existing example to copy. The current best guess (`extractOfferId()` in `supabase/functions/booking-hotel-search/index.ts`) tries `roomTypes[0].offerId` first, with a few fallback locations, and logs (`console.warn`, once per search request) the actual keys seen on the rate object whenever it can't find one — **check the edge function logs for that warning first** if `offerId` turns out to always be `null` in production.

- Everything degrades gracefully rather than guessing silently: a missing/wrong field name results in a logged warning and a `null`/failure value propagating forward, never a fabricated value, and never an unhandled throw.

**A payment-succeeded-but-supplier-reservation-failed booking is a real, currently possible state — manual recovery only.** There is no automated refund path (that would require testing against a live Paystack refund API this sandbox cannot verify). To find and handle these: query `bookings` where `details->>'supplierBookingFailed' = 'true'`, read `details->>'supplierBookingFailureReason'` for why, and if a refund is warranted, issue it manually via the Paystack dashboard. This is an accepted, documented gap, not something silently papered over.

### Real check-in date selection (booking modal)

The booking modal (`#stepNights`) now has a real `#checkinInput` (`type="date"`, next to the existing nights stepper) — check-out is always **derived** as check-in + nights (`computeCheckoutDate()` in `index.html`), never a second independently-editable field, so the user only ever sets one date plus a length of stay. It defaults to today+7 each time `openPackageModal()` opens (matching `booking-hotel-search`'s own default window, so nothing looks different until the user actually changes it) — there's no persisted "last search" to restore.

Because LiteAPI's rates are genuinely date-dependent (seasonality, weekday/weekend), changing the check-in date or the nights count on a **real** LiteAPI hotel (one with `activeHotel.offerId`/`activeHotel.liteApiId` set — see `adaptHotelSearchResult()`) triggers `repriceActiveHotel()` in `index.html`, which calls `booking-hotel-search`'s new single-hotel reprice mode (`{hotelId: activeHotel.liteApiId, checkin, checkout, count: 1}`) and, on success, overwrites **both** `activeHotel.pricePerNight` and `activeHotel.offerId` together (the offer is tied to specific dates — the old `offerId` is for the old dates and must never be reused for booking), then calls `updateTotal()` so the itemized breakdown reflects the real new price. **Mock/fallback hotels from `genHotels()` are unaffected** — no `offerId`/`liteApiId`, so `repriceActiveHotel()` returns immediately and they keep doing exactly what they always did: simple `pricePerNight * nights`.

**If repricing fails** (network error, LiteAPI error, or an empty/unpriced result) the book button (`#bookBtn`) is disabled and `#repriceStatus` shows "Couldn't confirm pricing for these dates — try different dates." — the user is **blocked from booking**, not shown a stale price silently. The book button is also disabled for the whole duration of an in-flight reprice call (not just on failure), so a booking can never go through using an `offerId` that was priced for different dates than what's currently selected. `bookBtn.onclick`'s payload to `create-booking-payment` now includes `details.checkin`/`details.checkout` (additive, alongside the existing `hotelName`/`flight`/`hotelTotal`/etc.) so the actual chosen travel dates are recorded on the booking.

**CONFIDENCE — same LOW-MODERATE/UNVERIFIED level as the rest of the LiteAPI integration.** The single-hotel reprice path reuses the exact same rate-fetching/`extractOfferId()` logic as the original list search (no new field-name guesses), but has not been exercised against a real LiteAPI response — whether `/v3.0/hotels/rates` genuinely returns a single hotel correctly for a one-element `hotelIds` array, and whether the price it returns truly reflects the requested dates, is unverified (no live network access to LiteAPI from this sandbox).

`get_wallet_balance` was removed from `concierge-chat`'s tool list (both the Claude-facing tool schema and the handler branch) since `wallets` is a dormant table post wallet-removal — Gabriella no longer claims it can check a wallet balance.

## Legal (`/legal/`)

`terms-of-service.md`, `privacy-policy.md`, and `refund-policy.md` are **AI-drafted starting points, not reviewed by a qualified attorney**, written to save drafting time — each has a prominent banner saying so and is full of `[bracketed]` placeholders (legal entity name, registration number, retention periods, etc.). They flag several points that specifically need South African legal review before publishing to real customers: CPA Section 51 liability-limitation restrictions, POPIA's Information Officer registration requirement and cross-border transfer rules (data is hosted in Supabase's `eu-west-1`/Ireland region), and how CPA cooling-off rights interact with dated travel bookings. Do not treat these as ready to publish.

They're linked from the UI as plain `<a target="_blank">` tags in two places: the profile overlay footer (`#profileOverlay`) and the auth overlay footer (`#authOverlay`, `.legalLinks`) — minimal, unobtrusive, not a dedicated legal/settings page.

## Architecture (all in `index.html`)

The app is a single-page state machine over a handful of full-screen overlay `<div>`s (`#globeUI`/`#globeCanvas`, `#mapView`, `#tripsView`, `#modalOverlay`, `#confirmOverlay`, `#profileOverlay`, `#onboardOverlay`, `#authOverlay`, plus the separately-injected `#gabrielPanel`), toggled via `display`/CSS classes by functions like `showMapView()`, `showTripsView()`, `openPackageModal()`, `openAuth()`. There's no framework/router — plain DOM queries and `on*` handlers, split across two inline `<script>` tags: the large main one (mock inventory through booking/trips/Gabriella) and a smaller one at the very end dedicated to the sign-in/up form and `sb.auth.onAuthStateChange`.

Reading top to bottom, the script is organized into these blocks (search for the `/* ==== ... ==== */` banner comments to jump between them):

- **Supabase client + auth gate** (`sb`, `currentUser`, `requireAuth`, `openAuth`/`closeAuth`) — declared right at the top of the main script, since almost everything below eventually depends on them.
- **Mock inventory** (`DESTINATIONS`, `genHotels`, `genFlights`, `genHotelName`) — all destinations/hotels/flights are procedurally generated by default. Comments note the data shapes are meant to be swappable for real GDS/hotel-supplier feeds without touching UI code — which is exactly what happened for hotels (see Flight/hotel edge functions above).
- **Hotel "photos"** (`buildSceneSVG`, `genericHotelSVG`, `photoTag`) — inline SVG illustrations as instant placeholders, with `photoTag()` layering on a real Unsplash photo (seeded by hotel id/scene) that falls back to a generated SVG on error.
- **Live hotel search** (`fetchRealHotels`, `adaptHotelSearchResult`) — calls the real `booking-hotel-search` edge function; see Flight/hotel edge functions above for the full contract and LiteAPI details.
- **Globe rendering** — a hand-rolled Canvas 2D 3D globe with no external map library: continents are a bit-packed base64 land/sea bitmap (`LAND_MASK_B64`, sampled via `sampleLand`), re-projected per-pixel each frame (`renderGlobeRaster`, `drawGlobe`, `animate`). Search box + Levenshtein fuzzy matching (`findPlace`, `findNearestPlace`) resolve free-text input to a `DESTINATIONS` entry or the nearest known place.
- **Destination map view** — switches to real tiles via Leaflet + leaflet.markercluster (`initLeafletMap`), with pins for hotels (real or generated), filter chips by property type, and a "search radius" ring.
- **Van Der Belt prioritized-availability engine** (`computePriorityScore`, `reprioritiseUnshown`) — a weighted scoring model (`MASTER_WEIGHTS`: customer/profit/service/supplier) that reorders the hotel results shown in the belt/map based on session-learned preferences (`notePreference`), while never showing zero results.
- **Search radius expansion** (`RADIUS_MIN/STEP/MAX`, `applyRadiusFilter`) — widens the search ring in steps until `MIN_RESULTS` hotels are found for the active filter.
- **Package builder modal** (`openPackageModal`, `renderFlights`, `updateTotal`, `bookBtn.onclick`) — lets the user pick nights/flight for a chosen hotel and computes the combined flight+stay+transfers total; payment is a single fixed path (card via Paystack, no chooser UI — see Payments above).
- **Itinerary** (`loadTrips`, `renderTripsList`, `cancelTrip`, `showBookingConfirmation`) — "My Trips" list, backed by Supabase (see Backend above) and refreshed after every mutation. `showBookingConfirmation(trip)` populates and shows the "Trip booked" confirmation screen from a real `bookings` row — used by `handlePaystackRedirect()` once a just-paid booking shows up. `cancelTrip()` is a best-effort real update (`bookings.details.cancelled = true` via the client, relying on RLS allowing an owner to update their own booking row) — added because the redesigned UI has a cancel button that the original app never had; this hasn't been verified against real RLS policies, so double-check it actually persists before relying on it.
- **Gabriella** (`initGabriel()` IIFE) — the AI concierge chat panel; see the AI concierge section above.
- **Profile** (`openProfile`/`saveProfileBtn`, `STORAGE_KEY='graba_state_v1'`) — a local-only contact card (name/email/phone) used to personalize the default lead-guest name and to build the WhatsApp booking-confirmation deep link (`whatsappConfirmBtn`). Not account data, not synced anywhere.
- **First-run onboarding** (`ONBOARD_STEPS`, `maybeShowOnboarding`) — a one-time three-step walkthrough, shown once per browser via `localStorage['graba_onboarded_v1']`.

### BRS traceability comments

Many blocks are annotated with comments referencing a Business Requirements Specification, e.g. `BRS Process 6.0`, `Value Streams M/N/O/P`, `Distinguishing Concept A/B/C/D/F/G`, `BRS §5.8.9-11`. These map specific code (wallet, split payment, search-radius expansion, prioritized availability, "never return zero availability") back to numbered requirements/sections in that external spec. Preserve these comments and their section references when touching the code they annotate — they're the link between behavior and the requirement it satisfies, not incidental documentation.

## State conventions worth knowing before editing

- All app state (`hotelPool`, `userPrefs`, `dismissedIds`, `myTrips`, etc.) is plain top-level `let`/`const` — no store/reducer pattern. Functions read and mutate these globals directly. `myTrips` specifically is a client-side cache refreshed wholesale from Supabase by `loadTrips()`, not the source of truth.
- Hotel "dismiss" (swipe away in the belt) doesn't delete the hotel — it reprioritizes it out of view (see `BRS Distinguishing Concepts B.2.2 / B.3.1` near `attachSwipeDismiss`). This is unrelated to and unaffected by the Supabase backend.
- Money is always South African Rand, formatted via `fmtR()`.
