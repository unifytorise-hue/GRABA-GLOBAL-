# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## This repo is now the only place this app is developed

`index.html` was rewritten by merging two previously-diverged copies of Graba:

1. **This repo's own `index.html`** (the live, deployed app) — plain vanilla JS with a real, working Supabase backend: real auth, a real Travel Wallet with Paystack top-ups, real trip/booking storage, a real Claude-powered AI concierge, and a hotel-search edge function.
2. **A separately Lovable-redesigned copy** of the same app, developed in the *other* repo `unifytorise-hue/graba-global` (lowercase, a Lovable/TanStack project) — much more polished UI (Playfair Display + Inter fonts, glow effects, mobile safe-area handling, a nicer multi-thread AI chat panel called **Gabriel**, a profile overlay, a first-run onboarding walkthrough, a "pack belt" destination carousel). Its auth/wallet/trips were **100% fake** — plain `localStorage`, a `Prototype · Demo Data` badge, no real backend at all.

Those two copies were merged in branch `claude/merge-lovable-ui`: the Lovable UI became the new `index.html`, with every piece of fake `localStorage` auth/wallet/trip logic ripped out and replaced with the real Supabase wiring from the original file (same tables, same RPCs, same edge functions, same `requireAuth()` gating pattern — see Backend below). **`unifytorise-hue/graba-global` (lowercase) is now superseded and frozen** — do not make product changes there; it was a UI-only prototype and has no real backend of its own. All future work, on both UI and backend, happens in this repo (`GRABA-GLOBAL-`, note the trailing dash) only. If you're ever unsure which repo to edit, it's this one.

## What this is

**Graba** ("Making Travel Simple") — a travel booking app: search a destination on an interactive 3D globe, browse hotels on a live map, bundle a flight + hotel + transfers into one package, and pay from an in-app "Travel Wallet," or ask **Gabriel** (the AI concierge) to help plan a trip. All markup, CSS, and JS still lives in one `index.html` (no build step, bundler, or framework), but auth and money live in a real backend rather than in-memory state — see "Backend" below. There is no test suite; the app is opened directly in a browser or served statically.

External dependencies loaded from CDN in `index.html`: Google Fonts (Playfair Display + Inter + Outfit), Leaflet 1.9.4 + leaflet.markercluster (destination map/tiles), the Supabase JS SDK v2, and per-hotel Unsplash/generated-SVG photos with graceful fallbacks (`photoTag()`).

Flight/hotel inventory is still procedurally generated mock data (see Architecture below). The plan is Sabre for flights (Bargain Finder Max for search, EnhancedAirBook for PNR creation) and Booking.com Partner/Demand API for hotels — chosen over routing hotels through Sabre too, since Sabre's hotel content/booking certification is a separate, weaker path than its flight product. Both are blocked on the respective partner accounts; see the edge functions below for the current seams.

## Backend (Supabase)

Auth, wallet balance, and bookings are backed by a dedicated Supabase project **"Graba Global"** (`mpkklamqfjwfuciuuxae`, eu-west-1) — separate from the unrelated AestheticFlow CRM project in the same org. Schema:

- `profiles` / `wallets` — 1:1 with `auth.users`, auto-created by an `on_auth_user_created` trigger (`handle_new_user()`) on signup.
- `wallet_transactions` — append-only ledger, one row per top-up/debit.
- `bookings` — one row per confirmed trip; flight/hotel/transfer details are stored in a `details jsonb` column shaped to match a real Sabre response, so swapping mock data for live Sabre PNR data later won't need a schema change.

All tables are RLS-scoped to `auth.uid()` (SELECT only from the client — no direct client-side balance/booking mutation). Balance changes and booking creation go through `security definer` RPCs instead (`anon`/`authenticated` execute is explicitly revoked by default and re-granted only where intended, since Postgres/Supabase grants `EXECUTE` to `anon` by default on new functions):

- `top_up_wallet(p_amount, p_note)` — **superseded, no longer client-callable.** Originally credited the caller's own wallet instantly with no real charge behind it (a demo placeholder); `authenticated`'s execute was revoked once Stripe payments were wired (see Payments below), since leaving it open would let anyone give themselves free wallet balance. Left in place (unused) rather than dropped, in case it's useful for a future admin tool.
- `admin_top_up_wallet(p_user_id, p_amount, p_note)` — the real replacement. `service_role`-only (not `authenticated`, not `anon`) — only callable by the `stripe-webhook` edge function after it has verified a real Stripe payment. See Payments below.
- `book_trip(p_destination_city, p_destination_country, p_traveller_name, p_nights, p_payment_method, p_total_amount, p_wallet_portion, p_details)` — debits the wallet portion (if any), checks sufficient balance, and inserts the booking, atomically. Generates its own `booking_ref` (`GRB-XXXXXXXX`).
- `add_insurance(p_booking_id)` — idempotently adds R150 to an existing booking's total and flags `details->>'insurance'`.

In `index.html`, `loadWalletAndTrips()` re-fetches wallet + transactions + bookings after every mutating RPC call and re-renders; there's no realtime subscription.

**Auth is on-demand, not a startup gate.** `#authOverlay` (email/password sign-in/sign-up, toggled via a `.on` CSS class rather than `display`) never shows automatically — anyone can browse the globe, search, view hotel/flight details, and open the package builder without an account. The single gate is `requireAuth(message)` (defined near the top of `index.html`'s main `<script>`, alongside the `sb` client): returns `true` immediately if `currentUser` is set, otherwise calls `openAuth(message)` to show `#authOverlay` with a contextual subtitle and returns `false`; callers just `return` when it's `false` and the user can retry the same action once signed in. It's used at exactly the points that need a real account: opening the wallet (`walletChipGlobe`/`walletChipMap`/`walletChipTrips`), opening trips (`tripsBtnGlobe`/`tripsBtnMap`), opening Gabriel (the AI concierge FAB), topping up the wallet, and — the one that actually matters — the "Book with Graba" button itself. `sb.auth.onAuthStateChange` (registered in the auth `<script>` block near the end of the file, next to the sign-in/up form) never force-shows the overlay on a null session; it just resets the wallet/trips display to empty. The header's `#authBtnGlobe` button (globe view, top-right) doubles as both the sign-in entry point and, once signed in, shows the account's first name and opens the same overlay in "account view" (name/email + sign out) — there's no separate burger menu in this UI.

Client-side `SUPABASE_URL`/`SUPABASE_ANON_KEY` in `index.html` are the anon key — safe to expose, since every table relies on RLS + the RPCs above rather than trusting the client.

### AI concierge — "Gabriel" (Claude)

`concierge-chat` is a chat-based AI travel concierge — a Claude tool-use loop (Anthropic Messages API) over Graba's own capabilities. In the UI it's branded **Gabriel**, with a floating action button, a multi-thread chat panel (`#gabrielPanel`/`#gabrielSheet`, `initGabriel()` IIFE near the end of the main `<script>`), and a thread list (`#gThreadsPane`/`#gThreadList`).

- **Two message histories per thread, deliberately kept separate**: `t.messages` (simple `{role, content:string}` pairs — used to render the chat UI and persisted to `localStorage` under `graba_gabriel_threads_v1`/`graba_gabriel_active_v1`, since which threads exist and their titles is just local UI bookkeeping, not account data) and `t.apiMessages` (the raw Claude Messages API history, which can include `tool_use`/`tool_result` content blocks) sent to and returned from `concierge-chat` so multi-turn tool use keeps working within a thread without leaking those blocks into the rendered chat. The actual chat completions always come from the real edge function — nothing about Gabriel's replies is mocked.
- Opening Gabriel (the FAB) is gated by `requireAuth()`; sending a message additionally checks for a live Supabase session (`sb.auth.getSession()`) since `concierge-chat` requires a bearer token.
- Not a streaming endpoint — `concierge-chat` returns a final `{reply, messages}` once Claude's tool-use rounds finish, so `send()` in the Gabriel IIFE shows a typing indicator and then renders the whole reply at once (there's no token-by-token stream to consume, despite the UI supporting one).
- **Tools exposed to Claude**: `search_flights` and `search_hotels` (server-to-server HTTP calls to the sibling `sabre-flight-search`/`booking-hotel-search` edge functions, using the service role key — they inherit those functions' own mock-data fallback behavior), `get_wallet_balance` (direct query), and `propose_booking`.
- **Guardrail — do not relax without discussion**: `propose_booking` is a pure pass-through that returns a structured proposal for the frontend to render; it does **not** call `book_trip`. Nothing in this function can spend money or create a real booking — the user still has to go through the existing package-builder modal and tap Confirm. This is deliberate: an LLM autonomously committing bookings/payments with no human confirmation step is not a safe default. The system prompt also explicitly tells Claude never to claim something is booked/charged.
- **Confidence**: HIGH on the Claude/Anthropic API shape itself (Anthropic's own documented API, unlike the Sabre/SerpApi/LiteAPI work), but this has **not** been tested end-to-end from this sandbox (no live Anthropic API access here either) — verify a real conversation once `ANTHROPIC_API_KEY` is set.
- Requires an `ANTHROPIC_API_KEY` edge function secret (console.anthropic.com) and, optionally, `CLAUDE_MODEL` to override the default (`claude-sonnet-5`).

### Payments (Paystack — active; Stripe is dormant)

Real money moves through **Paystack**, wired into the wallet top-up flow (`topUpWallet()` in `index.html`, both the preset chips and the custom-amount input; the post-redirect handler is `handlePaystackRedirect()`). The business went with Paystack over Stripe (better South African merchant support), so this superseded the earlier Stripe integration. Two edge functions:

- **`create-paystack-transaction`** (`verify_jwt: true`) — called by a signed-in user with `{amount}`; calls Paystack's `POST /transaction/initialize` (amount in kobo/cents, `ZAR`), tagging the transaction with `metadata.user_id`, and returns `{url: authorization_url}` for the frontend to redirect to. Requires `PAYSTACK_SECRET_KEY` and the user to have an email on file (Paystack requires one); falls back to a clear error if either is missing, since there's no safe fallback for real payments the way there is for search.
- **`paystack-webhook`** (`verify_jwt: false` — deliberately, Paystack calls it directly with no Supabase session, authenticating instead via HMAC-SHA512 of the raw body against the `x-paystack-signature` header, verified manually via Web Crypto since there's no Deno-friendly Paystack SDK used here) — the **only** path that actually credits a wallet. On `charge.success`, calls `admin_top_up_wallet` via the service role key using `metadata.user_id`. Deduplicates via a `paystack_events` table (transaction reference as primary key) since Paystack can redeliver webhooks.
- Requires edge function secrets: `PAYSTACK_SECRET_KEY` (both functions — also used to verify the webhook signature), optionally `SITE_URL` (defaults to the GitHub Pages URL) for the `callback_url`.
- **Redirect handling is different from Stripe**: Paystack uses a single `callback_url` for both success and failure (it appends `?trxref=...&reference=...`), unlike Stripe's separate success/cancel URLs — there's no explicit "cancelled" signal in the URL. `handlePaystackRedirect()` treats the presence of a `reference`/`trxref` query param as "a transaction was attempted," then polls `loadWalletAndTrips()` a few times (webhook delivery can lag the browser redirect) before opening the wallet view — if the payment actually failed, the balance just won't have moved.
- **Confidence**: moderate-high on Paystack's Transaction Initialize API and webhook signature scheme (well-documented, stable REST API) — higher than Sabre/Booking.com/SerpApi, but not at the same certainty level as the Claude/Anthropic integration. **Untested against a real Paystack account** from this environment (no live Paystack access here, and the business's Paystack activation/docs were still pending when this was written). Verify a real test-mode payment once `PAYSTACK_SECRET_KEY` is set and activated, and the webhook URL (`https://mpkklamqfjwfuciuuxae.supabase.co/functions/v1/paystack-webhook`) is registered in the Paystack dashboard.
- **Stripe's `create-checkout-session`/`stripe-webhook` functions and `stripe_events` table are still deployed but unused** — left in place rather than deleted in case Stripe is revisited later. Don't be confused by both existing; `topUpWallet()` only calls the Paystack one.

### Flight/hotel edge functions

Two edge functions are deployed on the Graba Global project as seams for real inventory. **Hotels are wired into the frontend; flights are not yet.** The hotel wiring degrades safely: `fetchRealHotels()` falls back to `genHotels()` mock data on any error, so a broken call just silently reverts to generated inventory rather than breaking the map view (this is the "never zero availability" principle — see below).

- **Hotels** (`fetchRealHotels()` / `adaptHotelSearchResult()` in `index.html`, called from `showMapView()` and `searchThisArea()`) — calls `booking-hotel-search` with `{query: "<city>, <country>", nights, count, lat, lon}` (`lat`/`lon` come straight from the matched `DESTINATIONS` entry, or the map's current center when panning), adapts its `{id,name,type,pricePerNight,rating,currency,lat,lon}` response shape into the full internal hotel shape (`x`/`y` percentage-space coords are derived by inverting `genHotels()`'s own lat/lon formula; `scene`, `rooms`, `supplierPriority`, `profitPct` have no real-world source so stay randomised even for real hotels — the last three are Graba's own invented business-logic weights, not something any real API would provide). On any error, empty result, or exception, `showMapView()` falls back to `genHotels(currentDestKey, 60)` mock data — the map view never breaks even if LiteAPI/the edge function is down or `LITEAPI_KEY` isn't set yet.
  - **Source: LiteAPI, not SerpApi.** The edge function's data source was switched from SerpApi's `google_hotels` engine (search/price-display only, scraped, never bookable) to **LiteAPI** (https://www.liteapi.travel/), a real bookable hotel-supply API. The frontend request/response contract is unchanged — this was a same-contract backend swap, the frontend didn't need to change field names.
  - The edge function source now lives in this repo at `supabase/functions/booking-hotel-search/index.ts` (previously it existed only as a deployed function, not version-controlled). It calls LiteAPI's `GET /v3.0/data/hotels` (by lat/lon, 30km radius, capped at 20 results) to list nearby properties, then `POST /v3.0/hotels/rates` (checkin = today+7 days, checkout = checkin+nights, matching the old SerpApi function's date defaults) to price them. A hotel is only included in the response if the rates call actually returned a price for it — the function never invents a `pricePerNight`, so if pricing fails for a hotel it's dropped rather than given a fake number.
  - Requires a `LITEAPI_KEY` edge function secret (read via `Deno.env.get("LITEAPI_KEY")`) — **not yet set**. Until it is, the function returns a clean `500 {error: "LITEAPI_KEY is not configured as an edge function secret"}`, and the frontend's existing fallback-to-mock-hotels behavior kicks in exactly as it did for the missing `SERPAPI_KEY` before — hotel search still "works" (shows plausible-looking hotels), it's just not real inventory yet. **Set `LITEAPI_KEY` before hotel search returns real, bookable results.**
  - `loadMoreBelt()`'s infinite-scroll padding still uses local mock data unconditionally (deliberate — avoids burning LiteAPI quota on scroll, and blends real initial results with the app's existing "never zero availability" padding).
- **Flights** (`sabre-flight-search`) — still not called by the frontend's booking flow at all; `openPackageModal`/`shuffleFlights` use local `genFlights()` directly (there's a separate, unrelated `fetchRealFlights()` in `index.html` that calls a Lovable-prototype-only `/api/public/flights` endpoint that doesn't exist in this static deployment — it fails silently and falls back to `genFlights()`, so it's harmless, but it's dead code pointing at nothing; wiring `sabre-flight-search` for real is still open). Wire it up once Sabre credentials are sorted and this has been tested for real (see the section below).

- **`sabre-flight-search`** (v6) — does the Sabre OAuth2 token exchange, then calls Bargain Finder Max. Both request and response are now built against verified real samples:
  - **Request** (`buildBFMRequest()`) — built against a *verified real* `OTA_AirLowFareSearchRQ` sample provided directly from Sabre's own docs.
  - **Response** (`parseBFMResponse()`) — built against a *verified real* `groupedItineraryResponse` sample (Sabre's modern JSON BFM response, not the older XML-derived `OTA_AirLowFareSearchRS` shape). Walks the reference-ID chain: `itineraryGroups[].itineraries[].legs[].ref` → `legDescs[]` → `schedules[].ref` → `scheduleDescs[]` for airline/time/duration/stops, and `pricingInformation[0].fare.totalFare` for price. Also checks `messages` for `severity: "Error"` entries and surfaces them. Pass `{"debug": true}` to always get the raw response back for comparison regardless.
  - **Known gap — currency**: the verified response sample prices in USD (`totalFare.currency`), but nothing in Graba's UI does currency conversion — `fmtR()` just prefixes with "R" unconditionally. Real Sabre prices will be silently mislabeled as ZAR until a conversion step is added. `FlightOption` now carries a `currency` field so the frontend can at least detect this once wired in.
  - **Token exchange encoding** (`getSabreToken()`) — `SABRE_CLIENT_ID`/`SABRE_CLIENT_SECRET` are the raw "User ID"/"Password" from Sabre Dev Studio's Applications page (format `V1:xxxxxxxxxxxx:DEVCENTER:EXT`). Standard OAuth2 Basic auth is a single `base64(id:secret)`, but Sabre has historically documented a Sabre-specific *double*-encoding scheme for this exact credential pair — recalled with moderate confidence, not verified. The function tries single-encoding first and automatically falls back to double-encoding if rejected; whichever worked is reported back as `tokenEncodingUsed`.
  - **Endpoint path** (`SABRE_BFM_PATH`) — still unverified, defaults to a guessed `/v2.3.0/shop/flights`; override via that secret once the real path is confirmed from Sabre Dev Studio docs.
  - Requires `SABRE_CLIENT_ID`, `SABRE_CLIENT_SECRET`, and `SABRE_PCC` (your account's Pseudo City Code) as edge function secrets — falls back to mock flights (`genFlights()`-shaped) if any are missing.
  - **Origin airport**: `index.html` now has a real "Flying from" `<select>` in the package builder modal (populated from `DESTINATIONS`, defaulting to Johannesburg/`JNB` — global `originCode`/`originCity`), so this is no longer nonexistent. It's just not plumbed into an actual `sabre-flight-search` call yet, since the frontend doesn't call that function at all (see above).
  - Booking a selected flight into a PNR would additionally need EnhancedAirBook, not yet stubbed anywhere.
- **`booking-hotel-search`** (despite the name, doesn't target Booking.com) — hotel search backed by **LiteAPI** (see above for the full description; source at `supabase/functions/booking-hotel-search/index.ts`). LiteAPI is bookable, unlike the SerpApi engine it replaced, but this function is still search/price-display only — it doesn't create a reservation itself. A genuine booking step (holding/confirming a LiteAPI rate) is a separate piece of future work. Pass `{"debug": true}` in the request body to get the raw LiteAPI list+rates responses back alongside the mapped hotels, to check/fix the mapping against real output. **`LITEAPI_KEY` must be set manually** (Supabase dashboard → Project Settings → Edge Functions → Secrets, or `supabase secrets set LITEAPI_KEY=...`) — no MCP tool here can set edge function secrets.

## Legal (`/legal/`)

## Architecture (all in `index.html`)

The app is a single-page state machine over a handful of full-screen overlay `<div>`s (`#globeUI`/`#globeCanvas`, `#mapView`, `#tripsView`, `#walletOverlay`, `#modalOverlay`, `#confirmOverlay`, `#profileOverlay`, `#onboardOverlay`, `#authOverlay`, plus the separately-injected `#gabrielPanel`), toggled via `display`/CSS classes by functions like `showMapView()`, `showTripsView()`, `openWallet()`, `openPackageModal()`, `openAuth()`. There's no framework/router — plain DOM queries and `on*` handlers, split across two inline `<script>` tags: the large main one (mock inventory through booking/wallet/trips/Gabriel) and a smaller one at the very end dedicated to the sign-in/up form and `sb.auth.onAuthStateChange`.

Reading top to bottom, the script is organized into these blocks (search for the `/* ==== ... ==== */` banner comments to jump between them):

- **Supabase client + auth gate** (`sb`, `currentUser`, `requireAuth`, `openAuth`/`closeAuth`) — declared right at the top of the main script, since almost everything below eventually depends on them.
- **Mock inventory** (`DESTINATIONS`, `genHotels`, `genFlights`, `genHotelName`) — all destinations/hotels/flights are procedurally generated by default. Comments note the data shapes are meant to be swappable for real GDS/hotel-supplier feeds without touching UI code — which is exactly what happened for hotels (see Flight/hotel edge functions above).
- **Hotel "photos"** (`buildSceneSVG`, `genericHotelSVG`, `photoTag`) — inline SVG illustrations as instant placeholders, with `photoTag()` layering on a real Unsplash photo (seeded by hotel id/scene) that falls back to a generated SVG on error.
- **Live hotel search** (`fetchRealHotels`, `adaptHotelSearchResult`) — calls the real `booking-hotel-search` edge function; see Flight/hotel edge functions above for the full contract and LiteAPI details.
- **Globe rendering** — a hand-rolled Canvas 2D 3D globe with no external map library: continents are a bit-packed base64 land/sea bitmap (`LAND_MASK_B64`, sampled via `sampleLand`), re-projected per-pixel each frame (`renderGlobeRaster`, `drawGlobe`, `animate`). Search box + Levenshtein fuzzy matching (`findPlace`, `findNearestPlace`) resolve free-text input to a `DESTINATIONS` entry or the nearest known place.
- **Destination map view** — switches to real tiles via Leaflet + leaflet.markercluster (`initLeafletMap`), with pins for hotels (real or generated), filter chips by property type, and a "search radius" ring.
- **Van Der Belt prioritized-availability engine** (`computePriorityScore`, `reprioritiseUnshown`) — a weighted scoring model (`MASTER_WEIGHTS`: customer/profit/service/supplier) that reorders the hotel results shown in the belt/map based on session-learned preferences (`notePreference`), while never showing zero results.
- **Search radius expansion** (`RADIUS_MIN/STEP/MAX`, `applyRadiusFilter`) — widens the search ring in steps until `MIN_RESULTS` hotels are found for the active filter.
- **Package builder modal** (`openPackageModal`, `renderFlights`, `updateTotal`, `renderPayOptions`) — lets the user pick nights/flight/payment method (Travel Wallet or card, with automatic split if the wallet balance falls short) for a chosen hotel and computes the combined flight+stay+transfers total.
- **Travel Wallet & itinerary** (`topUpWallet`, `loadWalletAndTrips`, `renderWalletView`, `renderTripsList`, `cancelTrip`) — wallet balance/transaction log and "My Trips" list, backed by Supabase (see Backend above) and refreshed after every mutation. `cancelTrip()` is a best-effort real update (`bookings.details.cancelled = true` via the client, relying on RLS allowing an owner to update their own booking row) — added because the redesigned UI has a cancel button that the original app never had; this hasn't been verified against real RLS policies, so double-check it actually persists before relying on it.
- **Gabriel** (`initGabriel()` IIFE) — the AI concierge chat panel; see the AI concierge section above.
- **Profile** (`openProfile`/`saveProfileBtn`, `STORAGE_KEY='graba_state_v1'`) — a local-only contact card (name/email/phone) used to personalize the default lead-guest name and to build the WhatsApp booking-confirmation deep link (`whatsappConfirmBtn`). Not account data, not synced anywhere.
- **First-run onboarding** (`ONBOARD_STEPS`, `maybeShowOnboarding`) — a one-time three-step walkthrough, shown once per browser via `localStorage['graba_onboarded_v1']`.

### BRS traceability comments

Many blocks are annotated with comments referencing a Business Requirements Specification, e.g. `BRS Process 6.0`, `Value Streams M/N/O/P`, `Distinguishing Concept A/B/C/D/F/G`, `BRS §5.8.9-11`. These map specific code (wallet, split payment, search-radius expansion, prioritized availability, "never return zero availability") back to numbered requirements/sections in that external spec. Preserve these comments and their section references when touching the code they annotate — they're the link between behavior and the requirement it satisfies, not incidental documentation.

## State conventions worth knowing before editing

- All app state (`hotelPool`, `walletBalance`, `userPrefs`, `dismissedIds`, `myTrips`, etc.) is plain top-level `let`/`const` — no store/reducer pattern. Functions read and mutate these globals directly. `walletBalance`/`walletTx`/`myTrips` specifically are a client-side cache refreshed wholesale from Supabase by `loadWalletAndTrips()`, not the source of truth.
- Hotel "dismiss" (swipe away in the belt) doesn't delete the hotel — it reprioritizes it out of view (see `BRS Distinguishing Concepts B.2.2 / B.3.1` near `attachSwipeDismiss`). This is unrelated to and unaffected by the Supabase backend.
- Money is always South African Rand, formatted via `fmtR()`.

## CRITICAL bug fix: CORS was silently breaking every real API call, project-wide

Discovered while live-testing `booking-hotel-search` after `LITEAPI_KEY` was finally set: the edge function logs showed every single request failing as `OPTIONS | 405`, before the search logic ever ran. Neither `booking-hotel-search` nor `create-paystack-transaction` (the two client-invoked functions this branch calls) had any CORS handling — no `OPTIONS` handler, no `Access-Control-Allow-*` headers. Since `supabase-js`'s `.functions.invoke()` always triggers a browser CORS preflight for cross-origin JSON POSTs (github.io calling supabase.co always counts), **every real hotel search and wallet top-up from the live site has been failing at the preflight stage this entire project**, regardless of whether `LITEAPI_KEY`/`PAYSTACK_SECRET_KEY` were ever set correctly. The mock-data fallback silently absorbed this for hotel search, which is why it went undetected — the app always *looked* like it was working.

Fixed by adding a shared `CORS_HEADERS` object to both functions: an `if (req.method === "OPTIONS") return new Response(null, { headers: CORS_HEADERS })` as the first line inside `Deno.serve`, and `...CORS_HEADERS` spread into every `Response`'s headers. Standard, well-established CORS handling — high confidence, unlike the LiteAPI/Sabre field-name uncertainties elsewhere in this file.

`create-paystack-transaction` was also added to this repo's `supabase/functions/` for the first time here (previously deployed-only, never version-controlled).

**If "the mock/fallback data keeps showing even though the secret is definitely set" ever comes up again, check CORS first.**
